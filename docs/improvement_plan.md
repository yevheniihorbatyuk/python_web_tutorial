# Пропозиції щодо подальших покращень

## 1. Централізація підключень до бази даних у продвинутих прикладах
- **Що відбувається зараз:** модуль `advanced_examples/patterns/02_repository_pattern.py` створює підключення через `psycopg2.connect(self.connection_string)` і вимагає ручного connection string у кожному прикладі.
- **Чому це проблема:** дублікація налаштувань БД та відсутність єдиного місця для оновлення конфігурації.
- **Що робити:** перевести модуль на `utils.db.DatabaseConfig` та `utils.db.get_cursor`, як це зроблено в `00_complete_tutorial.py`, щоб приклади автоматично підхоплювали `.env`.

## 2. Усунення ручних маніпуляцій з `sys.path`
- **Що відбувається зараз:** у файлах `advanced_examples/patterns/02_repository_pattern.py` та `advanced_examples/etl/01_async_etl_pipeline.py` додається корінь репозиторію через `sys.path.append(...)`.
- **Чому це проблема:** дублює код та приховує необхідність правильного пакування проєкту.
- **Що робити:** перетворити `python_web_tutorial` на інсталювальний пакет (``pyproject.toml`` + ``pip install -e .``) або принаймні винести логіку додавання шляху в окрему утиліту.

## 3. Автоматичні тести та перевірки
- **Що відбувається зараз:** у репозиторії відсутні smoke-тести для перевірки прикладів, а збірка завершується з помилкою на `python -m compileall` через `await` у `00_complete_tutorial.py`.
- **Чому це проблема:** важко перевірити, що навчальні скрипти продовжують працювати після змін.
- **Що робити:** додати мінімальний набір тестів (наприклад, ``pytest``) або окремі smoke-скрипти, які імпортують модулі й викликають ключові функції без реального підключення до БД (mock).

## 4. Стандартизація стилю коду
- **Що відбувається зараз:** змішані лапки, вручну відформатовані SQL, немає єдиного інструмента для lint/format.
- **Чому це проблема:** важко підтримувати однаковий стиль у матеріалах.
- **Що робити:** додати `ruff` або `black` до `requirements-dev.txt` та GitHub Actions для автоматичної перевірки стилю.

## 5. Покращення документації щодо середовища
- **Що відбувається зараз:** `00_complete_tutorial.py` містить текстові інструкції, але README не має короткого чекліста запуску Docker/PostgreSQL.
- **Що робити:** розширити `README.md` розділом «Підготовка середовища» з витягом ключових кроків та посиланням на `docker-compose.yml`.

## 6. Прибирання застарілих псевдонімів
- **Що відбувається зараз:** у `utils/helpers.py` залишились псевдоніми `DB_CONFIG` та `with_connection` тільки заради сумісності.
- **Що робити:** поступово замінити використання цих псевдонімів у всіх модулях, а тоді видалити їх, щоб API був прозорим.

## 7. Окремі дані для демонстрацій
- **Що відбувається зараз:** директорія `data/` містить CSV/SQL, але немає скрипта, який би гарантував оновлення цих файлів перед демонстрацією.
- **Що робити:** додати CLI-утиліту (наприклад, ``python -m tools.bootstrap_data``), яка ініціалізує дані в БД та перевіряє їхній стан.
